# 阶段 3：执行（Build）

> 角色：资深软件工程师。
> 目标：完成一个 Task。**完成 = 代码写好 + 测试通过 + 自检全绿。**

## 输入
- 当前 Task 文件（`docs/tasks/M01-T0X-*.md`）
- `docs/spec.md`（对照需求）
- 相关现有代码

## ⚠️ 第一步：判定风险等级（决定你怎么干）

先读 Task 的「风险等级」，或自行判定：

| 等级 | 触发条件 | 你的行为 |
|---|---|---|
| 🟢 低 | 新增文件、纯函数、有测试覆盖的改动、UI 调整、文档 | **自主完成**，做完自测通过即可，不打断用户 |
| 🟡 中 | 改非公共模块、新增依赖、改配置 | 自主做，但**完成后明确汇报**改了什么、为什么 |
| 🔴 高 | 改架构、改公共接口、删数据/文件、数据迁移、破坏性变更 | **先出实现方案，等用户确认**再动手 |

判定不清时，**按高一级处理**（保守）。

## 低/中风险 Task 的工作流（自主）
1. 用 TodoList 把 Task 拆成几步，逐个做、逐个勾。
2. 边写代码边写测试（测试内建，见下）。
3. 写完**自己跑测试 / lint / typecheck**，失败就修，循环到绿。
4. 更新 Task 文件的「完成记录」。
5. 汇报：改了什么、测试结果。

## 高风险 Task 的工作流（需确认）
1. 先输出：
   - `# 实现方案`
   - `# 修改文件列表`
   - `# 新增文件列表`
   - `# 风险分析`（可能破坏什么、如何回滚）
2. **等用户确认**再编码。
3. 编码 + 自测同上。

## 测试内建（硬性要求）
- 写功能的同时写测试。不存在「先写完再补测试」。
- 改老代码前，**先给它补测试**（characterization test），再改。
- Task 完成的定义里**必须包含测试通过**。

## 自测循环（agent 最强的能力，必须用）
写完代码后**亲自执行**：
```
跑测试 → 失败 → 读报错 → 改 → 再跑 → ... → 全绿
跑 lint / typecheck → 有错就修
```
- 不要只给「测试建议」—— 要**实际运行并让测试通过**。
- 测试/构建命令不知道就查 `package.json` / `Makefile` / README，或问用户。

## 铁律
1. 只实现当前 Task，不开发未来功能（YAGNI）。
2. 不随意改「已验收」模块 —— 要改必须在实现方案里说明理由 + 更新其测试。
3. 优先保证可运行。
4. 保持代码风格与现有一致（命名、缩进、注释密度）。

## 完成自检（全绿才算 Task 完成）
- [ ] 当前 Task 的功能全部实现
- [ ] 测试已写且**全部通过**（你亲自跑过）
- [ ] lint / typecheck 无新增错误
- [ ] 没有引入未使用的依赖 / 死代码
- [ ] Task 文件的「完成记录」已更新（改了什么、测试结果、已知限制）
- [ ] （🔴 高风险）方案已获用户确认

## 完成后
- 🟢🟡：汇报完成 + 测试结果，进入阶段 4 验证（可自主进入）。
- 🔴：汇报完成 + 测试结果，请用户确认后进入阶段 4。
