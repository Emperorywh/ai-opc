# 阶段 4：验证（Verify）

> 角色：Staff 工程师 + QA。
> 目标：验证当前 Task（或整个 Milestone）**确实达标**。发现问题就修，不只是报告。

## 输入
- 代码 + 测试（阶段 3 产出）
- Task / Milestone 的验收标准（`docs/tasks/`、`docs/milestones/`）

## 验证内容（全部实际执行，不要只看代码）
1. **跑全量测试**（不只是当前 Task 的）：确认没有回归。
2. **lint / typecheck / build**：全部通过。
3. **Code Review**（可派 subagent 并行做多个维度）：
   架构、类型安全、性能、重复逻辑、潜在 Bug、边界情况、安全、可维护性。
4. **对照验收标准**逐条核对。

## ⚠️ 发现问题 → 回路（关键）
验证不是终点，是**修复循环的入口**：

- **P0（必须修复：Bug、安全、回归、验收标准未达成）**
  → **回到阶段 3**，建子任务修复 → 自测 → 回阶段 4 再验证。
  → 不要把 P0 留给用户。
- **P1（建议修复）**
  → 能顺手修就修；修不了就记到 `docs/tasks/backlog.md`，标 tech-debt。
- **P2（优化建议）**
  → 记到 backlog，不在本阶段处理。

## 输出（写回 Task 文件的「验证记录」）
```
# 验证记录
## 测试结果      （全量测试是否通过，覆盖率变化）
## 静态检查      （lint / typecheck / build）
## Code Review   （P0 / P1 / P2 清单 + 是否已修）
## 验收标准核对  （逐条 ✓ / ✗）
## 结论          PASS / 回阶段3修复（列出 P0）
```

## 完成自检
- [ ] 全量测试通过（含回归）
- [ ] lint / typecheck / build 通过
- [ ] 所有 P0 已修复并复测
- [ ] 验收标准逐条达成
- [ ] 验证记录已写入 Task 文件

## 完成后
- PASS → 进入阶段 5 集成。
- 有 P0 → 已在回路里修完，再来自检。
