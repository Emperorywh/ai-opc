## 0. 路线图设计原则（如何被应用）

本路线图严格遵守 SPEC 提出的五条原则，这里给出**在本项目中的具体落点**，避免后续执行时原则被稀释：

1. **Vertical Slice（纵向切片，第一原则）** — 每个 Milestone 都是端到端可演示的闭环（数据→渲染→交互→屏幕），不按技术层切。哪怕 M1 只有地形，它也是"打开页面看到真实起伏的世界地图沙盘"这个完整产品能力，而非"Terrain 模块完成一半"。
2. **骨架先行（第二原则）** — M1 同时承担"项目骨架（config/data/state/scene 目录与基础设施）"与"最高风险验证"双重职责。复杂 Shader 在 M1 只用基础高度分层，留到 M2/M4 完善。
3. **分层引入复杂度（第三原则）** — M1 基础设施+数据，M2 视觉，M3 交互，M4 数据+标签，M5 体验。每个 Milestone 原则上只引入**一个新的复杂度来源**。
4. **风险优先（第四原则）** — `DEM Pipeline / Heightmap / Projection / GPU Terrain` 这条最高风险链路**全部压在 M1**。绝不在最后才发现 DEM 流程不可行。
5. **Milestone 自治（第五原则）** — 每个 Milestone 独立开发、独立运行、独立测试、独立验收、独立提交，禁止"完成一半等后续阶段"。

### 0.1 关键工程决策（路线图级，覆盖性默认）

| 编号 | 决策 | 理由 |
|---|---|---|
| R1 | **DEM 数据源可插拔 + 程序化合成占位 DEM（MVP 免 GDAL）** | 本地无 GDAL；Copernicus GLO-30 全量约 30GB+ 且需 GDAL 烘焙。**MVP（M1–M5）全程用合成 heightmap**（大陆轮廓 mask + simplex 噪声）跑通**渲染链路**（真正风险点），完全不依赖 GDAL、不下载真实数据；真实 DEM 推迟到 MVP 后，走**免 GDAL 路径**（见 §五）。合成版与真实版输出**同格式** `heightmap.png + normal.png + meta.json`，渲染层零改动即可替换。 |
| R2 | **`project(lon,lat)→(x,z)` 单一函数作为全局对齐契约** | 地形顶点/边界/河流/标签锚点必须经同一投影。M1 锁定此函数签名，后续所有子系统依赖它，杜绝错位返工。 |
| R3 | **CPU 精简高度查询表与 GPU heightmap 共享同一烘焙源** | 解决 SPEC §10「CPU/GPU 高度不一致」陷阱；河流采样、拾取高亮深度偏移、标签锚点 y 均查此表。 |
| R4 | **Milestone 验收以"截图 + 可运行 dev server + 关键断言"三件套为准** | Claude Code 无法人工肉眼判视觉，故每 Milestone 附明确的可编程验收点（dom 渲染、几何顶点数、无 console error、FPS 阈值）。 |
| R5 | **每个 Task = 一次 commit（中文 message）** | 对齐全局规则；commit 粒度 = Task 粒度，便于回滚与上下文恢复。 |

## 二、Milestone 详细设计

### M1 · 地形沙盘地基 🔴（最高风险验证）

#### 目标
跑通最高风险链路：**离线 DEM Pipeline → heightmap 纹理 → 投影 → GPU 顶点位移 → 屏幕地形**。同时搭建整个项目的目录骨架与基础设施层。

#### 用户价值（用户可见成果）
打开 `pnpm dev`，在倾斜视角下看到一张 2:1 的世界地图平面，具备真实三维地形起伏（喜马拉雅、安第斯、落基、青藏高原、格陵兰、南极冰盖轮廓由高程自然呈现），七大洲形状可辨认。

#### 实现范围（技术成果）
- **项目骨架**：目录结构（按 SPEC §4.1）、依赖安装（drei/proj4/zustand/troika/topojson-client/earcut/martinez-polygon-clipping 等）、Zustand store 骨架、config 四件套（projection/palette/camera/quality）。
- **数据 Pipeline**：`0-synthetic-dem.mjs`（**程序化合成 DEM**：大陆轮廓 mask + simplex 噪声 → `heightmap.png + normal.png + meta.json`）。**MVP 不依赖 GDAL、不下载真实数据**（R1）。脚本设计为数据源可插拔——输出接口固定，未来真实 DEM（§五路径 A/B）同格式替换、渲染层零改动。
- **投影契约**：`config/projection.ts` 的 `project(lon,lat)→(x,z)`（R2）。
- **数据层**：`data/assets.ts`（fetch+parse→texture/typed array）、`data/types.ts`、CPU 高度查询表（R3）。
- **GPU 地形**：`Terrain.tsx`（PlaneGeometry + 顶点位移）+ `terrainMaterial.ts`（自定义 ShaderMaterial：heightmap 位移 + **基础**高度分层上色，仅海岸→平原→山脉→雪线）。
- **静态倾斜相机 + 基础光照**（方向光 + 半球光，无 shadow map）。

#### 不包含内容
- ❌ 海洋、水彩噪声、软描边、坡度强调（→ M2）
- ❌ 相机交互/pan/zoom（→ M3，M1 仅固定视角）
- ❌ 任何标签/字体（→ M4）
- ❌ 真实 Copernicus 全量数据下载（R1：合成 DEM 先行，真实数据为独立增强任务，不阻塞验收）

#### 涉及目录
`package.json`、`vite.config.ts`、`src/config/*`、`src/data/*`、`src/state/*`、`src/three/{Scene,camera,terrain}/*`、`scripts/data-pipeline/0-synthetic-dem.mjs + lib/`、`public/data/{heightmap.png,normal.png,meta.json}`、`src/App.tsx`、`src/main.tsx`

#### 依赖关系
- 无前置 Milestone（起点）。
- 被后续**所有** Milestone 依赖（投影契约 + 数据层 + 地形是地基）。

#### 风险验证项（本 Milestone 的核心价值）
1. ✅ **渲染链路可行性**（核心风险）：合成 heightmap 能正确加载为可采样纹理、GPU 位移正确——这是本项目真正的风险点，合成 DEM 已足够验证。**真实 DEM 处理可行性**（纯 JS geotiff 路线）单独推迟到 MVP 后，不阻塞 M1 验收。
2. ✅ **Heightmap 16-bit 加载正确性**：three.js 把 16-bit PNG 正确加载为可采样的纹理（易踩坑：需确认 `THREE.DataTexture`/` UnsignedIntType`/`type` 设置）。
3. ✅ **GPU 顶点位移正确性**：顶点着色器采样 heightmap、按 §5.1 映射 `y`、×2.5 夸张后地形对齐无错位。
4. ✅ **Projection 契约**：`project()` 数值边界正确（lon∈[-180,180]→x∈[-1,1]，lat∈[-90,90]→z∈[-0.5,0.5]，向北 -z）。
5. ✅ **CPU/GPU 高度一致性**：精简查询表与 shader 位移同源（R3）。

#### 验收标准
- `pnpm dev` 启动无 console error，Canvas 内出现 2:1 倾斜地形。
- 顶点数符合配置（如 1024×512），地形有明显起伏（最大高差 > 0）。
- `project()` 单元测试通过（边界值 + 采样点）。
- 合成 DEM 与（若可得）小样例真实 DEM 均能渲染出可辨认的大陆轮廓。
- CPU 高度查询表与 shader 位移在同一点 y 值误差 < 1e-4。

#### Definition of Done
- 全部验收通过；`pnpm build` 成功；截图归档 `docs/screenshots/M1.png`；单次 git commit。

#### 推荐 Git Commit 粒度
按 Task 拆分（见第三节 Task 01–05），每个 Task 一次 commit。M1 收尾可加一次 `feat(M1): 地形沙盘地基闭环验收`。

#### Claude Code 开发边界
- ✅ 允许：`src/{config,data,state,three}/**`、`scripts/data-pipeline/**`、`public/data/**`、`package.json`、`vite.config.ts`、根配置。
- ⚠️ 受限：`src/three/{ocean,borders,rivers,labels,atmosphere,effects}/**` 本阶段**仅建空目录占位，不实现逻辑**。
- 🚫 禁止：`src/ui/**`（M1 不动 UI 层）。

### M2 · 海洋与水彩质感 🟡

#### 目标
引入**视觉层**首个复杂度：完整水彩地形 shader + 动漫海洋系统。地形从"高度分层"升级为"程序化水彩 + 软描边"，海洋成为有体积/流动/深浅的非纯色背景。

#### 用户价值
地图呈现明确的"动漫/图鉴"质感：地形有水彩颗粒感与柔和轮廓描边，海洋有柔和浪涌、近岸浅青绿远洋深青绿的层次、掠射角菲涅尔微光。

#### 实现范围
- `Ocean.tsx` + `oceanMaterial.ts`：Gerstner 波（3–5 个，质量档降级为正弦）+ 菲涅尔 `pow(1-dot(N,V),3)` + 深浅渐变（heightmap 水深采样）+ 时间驱动流动。
- **透明渲染顺序**：Terrain 先绘写深度，Ocean 后绘关深度写入（SPEC §4.3 修正点）。
- `terrainMaterial.ts` 完善：坡度强调（陡坡暖灰绿/缓坡草绿）、水彩噪声叠加、软描边轮廓（`1-dot(N,V)`）、海岸线等高线（shader 内 `fwidth` 抗锯齿，非额外几何）。
- `palette.ts` 完整接入（SPEC §2.1 全色值，S 降 15–25% 统一处理）。
- shader 复杂度开关位（为 M3 的 AdaptiveQuality 预留 uniform 钩子）。

#### 不包含内容
- ❌ 相机交互（→ M3）；海洋波数自动分档逻辑（M3 接入，M2 仅预留开关）。
- ❌ 标签/辉光/边界/河流。

#### 涉及目录
`src/three/ocean/**`、`src/three/terrain/**`（完善）、`src/config/palette.ts`、`src/three/Scene.tsx`（渲染顺序）。

#### 依赖关系
- 依赖 M1（地形地基、heightmap、palette config）。

#### 风险验证项
1. ✅ **透明物体排序正确**：海洋不遮挡地形，海岸过渡自然（验证 SPEC §4.3 修正）。
2. ✅ **Gerstner 波性能**：4K 下多波叠加 60fps（若不达标，标记待 M3 分档）。
3. ✅ **水彩 shader 美学达标**：低饱和、非写实、无硬边（截图与 SPEC §2.2 逐项比对）。

#### 验收标准
- 海洋透明后绘不穿地形；近岸浅、远洋深渐变可见。
- 地形坡度着色与软描边肉眼可辨，色值符合 §2.1（低饱和）。
- 海岸线为 shader 自动等高线（非额外几何，可通过关闭开关验证）。
- 桌面 60fps（M3 未接入前可临时硬编码中档配置）。

#### Definition of Done
验收通过 + build 成功 + 截图 `docs/screenshots/M2.png`。

#### 推荐 Git Commit 粒度
Task 06–08 各一次 commit。

#### Claude Code 开发边界
- ✅ 允许：`src/three/{ocean,terrain}/**`、`src/config/palette.ts`、`src/three/Scene.tsx`。
- 🚫 禁止：`src/data/**`（数据资产不动）、`src/three/{borders,rivers,labels,atmosphere}/**`、`src/ui/**`。

### M3 · 相机交互与自适应质量 🟡

#### 目标
引入**交互层 + 性能层**：受限的沙盘相机控制 + 自动性能分档。地图从"展示型"变为"探索型"。

#### 用户价值
可拖拽平移、滚轮/捏合缩放，但 pitch 锁定、pan 受限不会滑出地图迷路；低端设备或掉帧时自动降级（波数/标签密度/dpr），4K 不爆显存。

#### 实现范围
- `SandboxControls.tsx`（D13）：固定倾斜 pitch（~45°，可配 45–60°）、yaw 小范围、pan 边界 `x∈[-1.1,1.1],z∈[-0.6,0.6]`、zoom min/max、阻尼平滑。
- `InputAdapter` 抽象（D20）：鼠标 + 触控板（预留触屏接口，不实现）。
- `AdaptiveQuality.tsx`（D18）：滑动窗口 FPS 统计（~1s 评估）+ `useDetectGPU` → 三档（高/中/低）→ 控制 dpr、shader 复杂度开关、标签密度（标签位先占位）。跨阈值平滑切换 + 手动覆盖。
- `config/camera.ts`（约束）、`config/quality.ts`（分档参数）落地。
- Zustand store 接入 `qualityTier`、`cameraZoom` 切片。

#### 不包含内容
- ❌ 键盘无障碍快捷键（→ 可选，M3 不强求，预留）。
- ❌ 触屏真实实现（仅预留接口）。

#### 涉及目录
`src/three/camera/**`、`src/three/effects/AdaptiveQuality.tsx`、`src/config/{camera,quality}.ts`、`src/state/store.ts`、`src/hooks/**`（输入 hook）。

#### 依赖关系
- 依赖 M1（场景/相机挂点）、M2（shader 复杂度开关钩子）。

#### 风险验证项
1. ✅ **Adaptive Quality 分档平滑**：FPS 抖动时不频繁切换、不抖动。
2. ✅ **4K 不爆显存**：dpr 上限对 4K 有效。
3. ✅ **pan 边界有效**：目标点严格落在 §6.6 边界内。

#### 验收标准
- 拖拽平移、滚轮缩放流畅；pitch 无法翻转、pan 不出界（边界测试用例）。
- 模拟掉帧（如人为降帧）能触发降档；`qualityTier` 切换有阻尼。
- 4K 分辨率下 dpr 受控，显存与帧率达标。
- 手动覆盖分档生效。

#### Definition of Done
验收通过 + build + 截图（含分档对比）。

#### 推荐 Git Commit 粒度
Task 09–11 各一次。

#### Claude Code 开发边界
- ✅ 允许：`src/three/{camera,effects}/**`、`src/config/{camera,quality}.ts`、`src/state/**`、`src/hooks/**`。
- 🚫 禁止：`src/data/**`、`src/three/{terrain,ocean}/**` 的 shader 主体（仅可读取开关 uniform）、`src/ui/**`。

### M4 · 大洲标签与中文字体 🟡

#### 目标
引入**数据层（标签）+ 文本渲染层**：中文字体子集化 + 高性能 SDF 标签 + 优先级视口碰撞。地图获得"图鉴"信息层。

#### 用户价值
七大洲、四大洋以清晰中文标注，标签不重叠；缩放时标签密度联动（低 zoom 仅大洲/大洋）。

#### 实现范围
- **字体子集化 pipeline**：`scripts/font-subset/`（subset-font，仅提取所需字形 → `public/fonts/map-zh.woff2`，数 KB~数十 KB）。
- **标签数据 pipeline**：`4-labels-zh.mjs`（ISO_A3 join 中文名表 → `labels.json`：`{id,zhName,continent,lon,lat,priority,kind}`），M4 仅产出**大洲+大洋**子集。
- `LabelLayer.tsx`：troika-three-text SDF 文本（每标签独立 Text 实例），锚点 = `project(lon,lat)` + CPU 高度查询表 `y`。
- `collision.ts`：优先级视口 AABB 碰撞剔除（按 priority 贪心放置，相交则隐藏）。
- LOD 缩放联动（订阅 M3 的 `cameraZoom`）。

#### 不包含内容
- ❌ 国家标签（→ M8）。
- ❌ 标签选中态 DOM 面板（→ M8）。

#### 涉及目录
`scripts/font-subset/**`、`scripts/data-pipeline/4-labels-zh.mjs`、`public/fonts/map-zh.woff2`、`public/data/labels.json`、`src/three/labels/**`、`src/hooks/useLabelCollision.ts`、`src/data/{assets,types}.ts`（扩展 Label）。

#### 依赖关系
- 依赖 M1（投影契约、高度查询表）、M3（cameraZoom）。

#### 风险验证项
1. ✅ **字体子集化体积**：woff2 控制在数十 KB 内，无缺字。
2. ✅ **troika SDF 性能**：数十标签 + 4K 60fps。
3. ✅ **碰撞剔除有效性**：标签无重叠（截图 + AABB 断言）。

#### 验收标准
- 七大洲四大洋中文标签全部显示且不重叠。
- 字体文件体积达标（断言 < 100KB）。
- 缩放联动：zoom 变化标签显隐符合 LOD 规则。
- 无缺字（中文渲染正确）。

#### Definition of Done
验收通过 + build + 截图 `M4.png`。

#### 推荐 Git Commit 粒度
Task 12–15 各一次。

## 四、实施风险报告

### 1. 最容易导致返工的模块

| 模块 | 返工风险 | 原因 | 缓解（已嵌入路线图） |

| **Projection 契约** | 🔴 极高 | 若各子系统各自实现坐标映射，切换投影（M9）时全部要改，且对齐 bug 难排查 | M1 锁定 `project()` 单一契约（R2）；M9 验证"渲染层零改动" |

## 五、真实 DEM 数据获取（MVP 后，免 GDAL）

> MVP（M1–M5）使用合成 DEM，无需本节。MVP 完成后若要真实地形，**无需安装 GDAL**，按省事程度选路径。所有路径输出**同格式** `heightmap.png`，直接替换 `public/data/heightmap.png` 并重生成 `normal.png`/`meta.json`，渲染层零改动。

| 路径 | 说明 | 省事度 | 来源 |
|---|---|---|---|
| **A. 现成全球 heightmap PNG** ⭐ | 直接下载已处理好的 equirectangular 高程图，扔进 `public/data/`，**绕过整个 pipeline**。先试这条 | 极高 | [Wikimedia World elevation map](https://commons.wikimedia.org/wiki/File:World_elevation_map.png)（equirect，1.8km/px）· [manticorp 16-bit 生成器](https://manticorp.github.io/unrealheightmap/)（浏览器生成 16-bit PNG，[GitHub](https://github.com/manticorp/unrealheightmap)）· [Solar System Scope textures](https://www.solarsystemscope.com/textures/) |
| **B. 纯 JS 读真实 Copernicus** | SPEC 原汁原味 Copernicus GLO-30，但用 `geotiff` npm 包代替 GDAL。AWS 上是匿名可读 COG 瓦片，读取→降采样→写 PNG。**取代原 GDAL 链路，全项目零 GDAL 依赖** | 中 | [AWS Open Data Copernicus DEM](https://registry.opendata.aws/copernicus-dem/) · [S3 bucket（匿名读）](https://copernicus-dem-30m.s3.amazonaws.com/readme.html) · [geotiff.js](https://geotiffjs.github.io/geotiff.js/) / [@developmentseed/geotiff](https://www.npmjs.com/package/@developmentseed/geotiff) |
| **C. 在线服务按区域下载** | OpenTopography 在 Web 上按经纬框下载已处理 GeoTIFF（仍需路径 B 的 JS 转 PNG） | 中 | [OpenTopography Copernicus GLO-30](https://portal.opentopography.org/raster?opentopoID=OTSDEM.032021.4326.3) |

**推荐顺序**：A（5 分钟看美学够不够）→ 不够再 B（Copernicus 精度）。装 GDAL（含 Docker 版）可永久不做。

> 路径 B 实现要点：把本路线图原 `1-dem-heightmap.mjs` 的 GDAL 链路替换为纯 JS——`@developmentseed/geotiff` 按需读 COG overview → 降采样到目标分辨率（如 8192×4096）→ 归一化高程到 16-bit → `pngjs`/`sharp` 写 PNG。南极精度不足处按 SPEC 用 REMA 补。
